CAS and JWT

CAS和JWT

CAS概念：

TGC:Ticket Granting Cookie

TGT:Ticket Granting Ticket

ST:Service Ticket

CAS center认证成功后，会创建一个Session，同时向浏览器返回一个名称为CASTGC的cookie，这个cookie中包含可以确定对应Session的TGT，同时浏览器重定向到之前请求的路径，包含唯一参数ST

应用App1对应一个ST，跳转到APP2时根据传递的TGC会找到对应的Session，会为APP2产生新的ST

第一次访问带有CAS客户端的系统会有3次请求：

1.直接访问地址

2.带有ST作为参数访问地址：会在客户端生成Session

3.带有JSESSIONID的cookie访问地址

cas无法避免csrf，引入jwt；token存储在localStorage中会带来xss攻击，为了避免xss，jwt的token有效时间要短，即使暴露危险性也很小

cas是为了实现统一登陆，统一登陆基础上的安全控制无法避免CSRF，而jwt才是获得访问权限的通用安全控制手段，要想访问必须有jwt

当前方式：

1.没有jwt时，先被cas拦截；有cas，需要校验jwt；有jwt，跳过cas

2.前端访问后台，需要jwt

​ 对外提供A系统接口，通过A系统的后台访问B系统的后台，A系统需要去获取token。通过传给A系统token，如果A系统执行时间过长，会出现访问B系统时token过期的情况。

3.第一次访问的时候，由于服务端无法区分界面访问还是接口访问，如果不提供任何凭证，会被cas拦截，但是如果提供了jwt，则会跳过cas

如果能够区分界面访问还是接口访问（最佳实践），

1.界面访问：先被cas拦截，cas通过后需要进行jwt校验，避免CSRF

2.接口访问：先被jwt拦截，jwt通过则跳过cas

如果能够区分界面访问还是接口访问，

1.界面访问：先被cas拦截，cas通过后需要进行jwt校验，避免CSRF

2.接口访问：先被jwt拦截，jwt通过则跳过cas

如果到每一个项目里面去校验cas，session只存在于其中某一个微服务模块，则会产生session共享问题

如果没有jwt，验证不通过，返回jwt不存在信息；如果有jwt，校验jwt，jwt通过并且cookie有效，则继续进行，否则跳转到cas

smartpole当前实现：

没有jwt，转到登录界面，登录成功后会刷新token，之后靠验证jwt来验证权限

authentication

1.JWT 是token format，OAuth是认证协议，可以使用JWT作为token

JWT：无法作废已颁布的令牌

2.

XSS:cross site scripting 跨站脚本攻击

如果将JWT token存放到LocalStorage中，恶意脚本会将token取出

防御：一定要在后台做检查，一般前后台都做检查，过滤特殊字符

CSRF:cross site request forgery 跨站请求伪造

如果使用cookie的话，恶意网站上发起请求时，cookie会自动添加

防御：验证码、referer检查、添加自定义token属性并验证

3.cookie

添加HttpOnly=true，则cookie不能被js获取到

添加Secure=true，则cookie只能使用https协议发送

4.

客户端的SingleSignOutFilter根据cas center回调时传递的ticket，删除sessionMappingStorage（存在于客户端的一个Map）中的session

5.临时cookie：Session cookie、会话cookie

不设置过期时间，表示cookie生命周期为浏览器会话期间，只要关闭浏览器窗口，cookie就消失了

6.永久cookie：Persistent cookie

设置了过期时间，浏览器会把cookie存储在硬盘上，关闭后再次打开浏览器，这些cookie依然有效直到超过设定的过期时间

7.Set-Cookie：包含在响应请求的头部中，用于在客户端创建一个Cookie

Cookie：由客户端发送，包含在HTTP请求的头部中

8.常见csrf_token的两种方式：

(1)Synchronizer Token Pattern(STP)

使用表单的隐藏域hidden，服务器端对每次请求都会更换csrf_token

(2)Cookie-to-header token：Double Submit Cookie

服务端为每个session生成一个csrf_token，保存到Set-Cookie中，客户端在每次请求时Cookie中的csrf_token设置到http request header中的X-CSRF_Token属性中

9.无法获取首次页面加载的response

10.浏览器同源策略(Same Origin Policy)，阻止恶意网站CSRF读取cookie中的值

但是如果HttpOnly为false，XSS攻击可以越过同源策略，将cookie取出来存储到txt文件中

osgi

osgi：Open Service Gateway Initiative

1.Eclipse集成了osgi容器Equinox

2.导出包：导出接口，会将接口的实现注册到容器

导入包：导入接口，根据接口再容器中寻找实现

3.问题：

不管存不存在消费端，接口的实现都会存在。某些service接口的实现比较耗费资源，最好是真正使用的 时候创建，不用的时候销毁

改进方法：

使用ServiceFactory接口的实现代替service接口的实现到osgi容器中去注册，只有其他bundle请求该服务时（根据接口名称获取服务），才会由ServiceFactory实现类来处理请求并返回一个新的service接口实现。

4.services跟踪

继承ServiceTracker类后，将service接口名传入构造函数，当bundle注册或取消注册service接口时，ServiceTracker的addingService方法和removeService方法会被调用

5.同一个接口下有多个实现注册，如何处理？

根据service的等级，在Service注册时的属性中可指定SERVICE_RANKING

等级也相同，看谁的PID小

6.每个bundle定义了自己的classloader

7.

注入bean的方式通过Hessian进行远程服务调用，注入的bean可以配置ServiceUrl(http格式的地址)和ServiceInterface
